# StartGrowUp
> Репозиторий содержит определения Protocol Buffers (`.proto`) и сгенерированный код для взаимодействия между сервисами.

## О проекте

Данный репозиторий является единым источником истины (Single Source of Truth) для API-контрактов сервиса.

В репозитории хранятся:

* определения Protocol Buffers (`.proto`);
* gRPC-сервисы;
* общие сообщения и структуры данных;
* сгенерированный код для поддерживаемых языков программирования;
* конфигурация генерации и проверки protobuf-контрактов.

Использование единого репозитория позволяет всем сервисам работать с одинаковыми версиями контрактов и исключает необходимость самостоятельной генерации protobuf-кода в каждом проекте.

# Версионирование

Версии API следуют принципам **Semantic Versioning**.

```
service/MAJOR.MINOR.PATCH
```

* **MAJOR** — изменения, нарушающие обратную совместимость;
* **MINOR** — добавление нового функционала без нарушения совместимости;
* **PATCH** — исправления ошибок, обновление документации или генерации без изменения API.

---

# Версии API

Каждая публичная версия API располагается в собственной директории.

Пример:

```text
proto/
└── service/
    ├── v1/
    ├── v2/
    └── v3/
```

Изменения, нарушающие совместимость, должны публиковаться только в новой версии API.

---

# Генерация кода

После изменения protobuf-файлов необходимо выполнить генерацию исходного кода.

Например:

```bash
make generate
```

или

```bash
buf generate
```

> **Важно:** не редактируйте файлы, находящиеся в директории `gen/`, вручную. Они полностью пересоздаются при каждой генерации.

---

# Проверка protobuf-контрактов

Перед созданием Pull Request рекомендуется выполнить проверку проекта.

Проверка оформления:

```bash
buf lint
```

Проверка на наличие несовместимых изменений:

```bash
buf breaking
```

---

# Использование

Подключите сгенерированный пакет в своем проекте.

Пример для Go:

```go
import pb "github.com/company/service-api/gen/go/service/v1"
```

Пример для Python:

```python
from service.v1 import service_pb2
```

# Обратная совместимость

Допустимые изменения:

* добавление новых сообщений;
* добавление новых RPC-методов;
* добавление новых полей;
* добавление необязательных полей.

Недопустимые изменения:

* удаление существующих полей;
* изменение типов полей;
* изменение номеров полей;
* повторное использование удалённых номеров;
* удаление сервисов;
* изменение protobuf-пакетов.

При необходимости подобных изменений должна быть создана новая версия API.

---

# Процесс разработки

При внесении изменений рекомендуется придерживаться следующего порядка действий:

1. Внести изменения в `.proto`-файлы.
2. Проверить корректность оформления (`buf lint`).
3. Проверить отсутствие несовместимых изменений (`buf breaking`).
4. Выполнить генерацию кода.
5. Убедиться, что все изменения попали в репозиторий.
6. Создать Pull Request.

---

# Непрерывная интеграция (CI)

При каждом Pull Request автоматически выполняются:

* проверка protobuf-файлов;
* проверка форматирования;
* генерация исходного кода;
* проверка обратной совместимости;
* контроль актуальности сгенерированного кода.

Pull Request не может быть объединён с основной веткой до успешного прохождения всех проверок.

# Основные правила

При работе с репозиторием необходимо соблюдать следующие требования:

* не изменять сгенерированный код вручную;
* поддерживать обратную совместимость API;
* документировать новые сообщения и сервисы;
* использовать общие protobuf-структуры при возможности;
* соблюдать соглашения по оформлению protobuf-файлов.

Подробные требования описаны в документах:

* `CONTRIBUTING.md`
* `STYLEGUIDE.md`
